10 - Proxy Services

Proxy Server Fundamentals

Source: https://github.com/zonzorp/NETS1037/blob/main/Presentations/NETS1037-07-ProxyServices.pdf
A proxy server can be used for several distinct purposes:

  • to cache web resources locally for performance or data transfer reduction (basic, most
    common setup)
  • to provide controlled access to protected resources (reverse proxy scenario)
  • to conceal the identity of an end-user's host from the target server (the tor
    network uses a network of proxies to hide users)
  • to restrict network traffic to specific choke points for traffic measurement and
    management purposes
  • to modify or reroute traffic before delivering it to clients or servers (advanced
    proxy configurations)

source: https://www.privateproxyguide.com/using-a-proxy-server-to-monitor-and-analyze-outgoing-traffic/
Different proxy roles map to different outcomes:

Proxy Type Position Primary Use Traffic Typical Tools
Forward proxy Client → Proxy → Internet Egress control, logging, content filtering HTTP/HTTPS (CONNECT), sometimes FTP Squid, Blue Coat, Zscaler, Envoy
Reverse proxy Internet → Proxy → Origin WAF, TLS offload, load balancing Inbound app traffic Nginx, HAProxy, Apache, Envoy
Transparent proxy Intercepts without client config Campus/branch enforcement HTTP/HTTPS (with bump/splice) Squid (TPROXY), policy-based routing
SOCKS proxy Client-configured Generic TCP/UDP tunneling Multi-protocol SOCKS5 (ssh -D), Dante

A forward proxy is an effective tool for logging and monitoring outbound traffic. Clients can connect to the proxy explicitly through browser or operating system proxy settings, or through automatic configuration using PAC/WPAD. In an enterprise network, traffic can also be transparently redirected at the network layer (L3/L4) using advanced switches and routers (e.g., policy-based routing) or proxy software (e.g., Squid with TPROXY), without requiring any client-side configuration.

Source: Wikipedia Article on WPAD:
PAC (Proxy Auto-Config) and WPAD (Web Proxy Auto-Discovery Protocol) are methods used to automatically configure proxy settings for web browsers:

  • Proxy auto-config (PAC) standard: create and publish one central proxy configuration file.
  • Web Proxy Auto-Discovery Protocol (WPAD) standard: ensure that an organization's browsers will find this file without manual configuration.

Caching

Another use case for web (HTTP/HTTPS) proxy servers is caching. Caching locally stores frequently accessed content, such as web pages, files, images, or other resources served from the internet, and subsequently serves the cached content to clients to reduce latency, improve transfer speeds, and conserve internet bandwidth.

The cache hit ratio is the proportion of client requests served from the cache compared to those forwarded to the internet source. A cache hit generally means that content is served instantly from local storage without routing the request to the origin server or downloading it again from the internet. This function has traditionally been used to improve performance on slower internet connections.

As another use-case example, caching is valuable in large network environments where many systems need to download the same files, such as software patches or updates. With local caching, files are downloaded from the internet only once, and subsequent client requests are served over the local network.

Some proxy software provides more control over caching than others. Depending on the software, various aspects of caching can be configured:

  • What to cache, such as HTML pages, images, videos, CSS files, JavaScript files, or other types of content
    • This concept is similar to browser-based web caching, except it applies network-wide
  • Retention policies and cache invalidation, including TTL (Time to Live expiration), LRU (Least Recently Used) eviction, storage quotas, and cache purging
  • Cache performance optimization, such as distributed caching, compression, and automated responses based on monitoring metrics (e.g., adjusting TTL or triggering cache purges based on cache hit/miss ratios)c.

Caching servers must be sized appropriately, with adequate memory, CPU, and storage resources, so they do not introduce bottlenecks in network performance.

A CDN (Content Delivery Network) can be categorized as a set of gateway cache servers (each gateway consisting of a cluster of servers depending on the scale of the CDN). These gateway caches sit between the origin server and the internet. Responses to clients are served by the CDN based on geographic location, latency, and other bandwidth considerations, with requests forwarded to the origin server only in the case of a cache miss.

Web admins can use specific HTTP headers that can inform the cache servers about some aspects of caching for their content:

  • The HTTP Expires header specifies how long the response is considered "fresh" (after which it becomes "stale")
  • The HTTP Cache-Control header allows finer control through directives such as max-age, max-stale, min-fresh, no-cache, and no-store

For more information, see documentation such as the one from Mozilla Developer Documentation for Cache-Control header.

Using a Proxy Server to Monitor and Analyze Outgoing Traffic

source: https://www.privateproxyguide.com/using-a-proxy-server-to-monitor-and-analyze-outgoing-traffic/
Deploying a forward proxy for egress inspection provides network and security teams with precise visibility into how internal clients interact with the internet. When properly implemented, a proxy becomes a centralized, policy-enforcing control point: it routes, observes, logs, and can optionally modify traffic before it leaves the network.

A thoughtfully configured forward proxy delivers granular visibility, policy enforcement, and actionable telemetry for all outbound web traffic. With Squid (or an equivalent secure web gateway), structured logging (e.g., JSON), and a modern analytics stack (such as Elastic/Logstash, Prometheus/Grafana), organizations can detect data exfiltration, malware command-and-control (C2) activity, shadow IT usage, and misconfigurations early, while preserving performance and user experience.

Before installing software, select a deployment pattern that matches your network topology and risk model.

Deployment Pattern How Clients Use It Identity/Policy Pros Cons
Explicit proxy Proxy host:port set in OS/browser or via PAC/WPAD Per-user (Kerberos/NTLM), per-subnet Best logging fidelity; easiest to debug Requires endpoint config hygiene
Transparent proxy L3/L4 redirect (policy-based routing, WCCP, TPROXY) Per-subnet or inline SSO No client config, full coverage HTTPS inspection is complex; HSTS/pinning issues
Hybrid Explicit for managed devices, transparent for guests/IoT Directory + subnet tags Coverage + control Two modes to operate/support

Recommendation: Start with explicit proxy via PAC/WPAD for corporate devices (enables identity-aware logging), then add transparent interception only where you can’t guarantee configuration (guest/IoT VLANs).

Squid

Source: https://github.com/zonzorp/NETS1037/blob/main/Presentations/NETS1037-07-ProxyServices.pdf
Squid is a proxy server which has been around since the mid 1990's (squid-cache.org). OPNSense, pfsense, and others have the ability to run squid as part of the gateway. Squid can also be installed on any Windows, Linux, or Unix server to provide proxy services. Other extensions and software can provide additional functionality (For example, ClamAV can be used to add signature-based anti-virus scanning). Consider blocking outbound direct traffic from the LAN to HTTP on the internet to force internal hosts to use the proxy for HTTP to the web. Squid is licensed under GNU GPL.

  • On Ubuntu and Debian, installation is simple from repositories: sudo apt install squid
  • The main configuration file is under /etc/squid3/squid.conf, which also contains the documentation.
  • Most default configurations are enough to start a simple HTTP caching proxy, but you must provide the acl and http_access settings.
  • The default port is set to 3128, but many browsers default to 8080, so the port number is a configuration to be aware of on the server and the clients
    • Port 8080 is used for content control software such as dansguardian, or e2guardian
  • Squid includes some CGI code for accessing cache management information via Web-UI:
    • Enable access to the web-UI by adding a line in squid.conf for http_access allow localnet manager
    • Some basic info can be retrieved by using the menu at http://proxyhost:3128/squid-internal-mgr/menu and replacing the word menu in that URL with a choice from the page retrieved by the menu URL
  • Squid keeps logs in /var/log/squid3
    • access.log shows traffic served
    • cache.log gives cache management details

HTTP vs. HTTPS Inspection

Since most, if not all, web traffic is encrypted using HTTPS, a forward proxy can only inspect a limited set of metadata, such as SNI (Server Name Indication), client fingerprint (using methods such as JA3 and JA4), and cert issuer. However, after the TLS handshake is complete, the traffic contents and the exact URLs and paths are no longer visible.

Inspecting traffic without breaking HTTPS is still useful. The following are examples of the level of visibility that a simple forward or transparent proxy provides:

  • Top destinations by domain or SNI
    • number of requests per client
    • users/client hitting blocked categories
  • TLS posture (cipher versions used)
    • Use of other/older TLS protocols is referred to as "TLS Posture Drift".
  • JA3 and JA4 client fingerprinting
    • JA3 = MD5 hash of (TLSVersion, Ciphers, Extensions, EllipticCurves, EllipticCurvePointFormats) from each client Hello TLS packet = 32 characters
    • JA4 = Protocol_CipherCount+CipherList_ExtensionCount+ExtensionList_SignatureAlgorithms
    • JA3 and JA4 Fingerprints are leveraged for network-based threat detection, such as known malicious clients (botnets, malware, unauthorized tools and software, etc), or can be combined with domain information for Indicators of Compromise (IoC) analysis.
  • Excessive client requests, or excessive 400 and 500 errors

source: https://docs.mitmproxy.org/stable/concepts/how-mitmproxy-works/
Inspecting the content of HTTPS requires an MITM (Man-In-The-Middle) proxy, with an internally trusted CA added to every client OS and browser. The MITM proxy will intercept every TLS handshake, impersonate the destination server, and take over the HTTPS communications.

  1. The client makes a connection to the MITM proxy and issues an HTTP CONNECT request.
  2. MITM proxy responds with a 200 Connection Established, as if it has set up the CONNECT pipe.
  3. The client believes it’s talking to the remote server and initiates the TLS connection. It uses SNI to indicate the hostname it is connecting to.
  4. MITM proxy connects to the destination server and establishes a TLS connection using the SNI indicated by the client.
  5. The server responds with the matching certificate, which contains the CN and SAN values needed to generate the interception certificate.
  6. MITM proxy generates the interception cert (signs its own cert using the trusted CA private key) and continues the client TLS handshake paused in step 3 using the interception cert.
  7. The client sends the request over the established TLS connection.
  8. MITM proxy passes the request on to the server over the TLS connection initiated in step 4.

The MITM proxy now has access to all unencrypted traffic and can perform inspection.

It is noteworthy to know that HTTPS can also be intercepted when using a transparent proxy, where a router is involved in forwarding connections. This requires tighter integration between the router and the proxy software since the proxy will need to know the SNI from the initial request, which was sent to the router by the client before the redirection. It is also important to note that the HTTPS MITM is generally not possible unless a matching trusted CA certificate is installed on all the clients.

Proxy Uses and Abuses

We have discussed ethical and legal considerations for logging and monitoring in earlier course content. Proxy servers are extremely useful because they provide application-layer logging and monitoring and support a wide range of security functions, from content filtering to threat detection and response. However, any tool can also be subject to abuse. Here are some (non-comprehensive) considerations when designing systems or deploying proxies:

  • Interception of encrypted traffic may be illegal in some jurisdictions, while in others it may be required by law.
    • For example, see the Wikipedia article on Kazakhstan man-in-the-middle attacks, which describes the government's attempts to require the installation of a government-issued root CA certificate by users
      • Later on, the Kazakh government unsuccessfully attempted to add the state-issued root CA certificate to Microsoft, Google, Apple and Mozilla root CA stores (in 2019 and again in 2020)
      • Many countries, such as Russia, China, Iran, and others, filter internet traffic in various ways, and the use of certain technologies is banned or restricted
    • In other jurisdictions, interception of encrypted traffic may be governed by wiretapping, unlawful interception, or data protection laws (e.g., GDPR, Canada PIPEDA, etc.)
  • A user’s “expectation of privacy” varies depending on jurisdiction and context. Organizational policies should explicitly define the level of privacy afforded to employees and other users of the organization’s information systems to ensure transparency. Interception of user communications may require awareness and explicit consent.
    • For example, a public or guest Wi‑Fi network may require users to accept terms of use before gaining access.
    • Consider how user PII or other sensitive information will be handled if it is captured in logs.
  • There are important distinctions between managed devices (e.g., corporate-owned devices) and user-owned (BYOD) devices.
  • Who has access to logged data:
    • Requirements for the implementation of the principle of least privilege
    • Definition of roles: who can access logs, who audits access, and what the consequences of non-compliance are
  • Breaking the terms of service for external service providers and website owners
    • The terms of use/service of internet/facing systems may prohibit bypassing geo-restrictions. If user traffic is routed through a remote proxy, it may obscure the user’s true location and could result in consequences
      • The consequences range from the proxy server’s public IP address being flagged or blocklisted to potential legal liability
