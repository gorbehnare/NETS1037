04 - Log Analysis

What is Log Analysis

Log management collects logs from various sources and stores them. Log analysis is the process of examining the log data. Logs contain records of system activities, and examining logs provides valuable insights into system performance, potential for optimization, potential security issues, or other failures. Log analysis can be performed manually by reviewing and correlating events based on time and date or other factors. For a very large volume of logs, big data analytics and visualization techniques can be used. Some log analysis tools may automate some tasks and highlight issues to make log analysis faster or more efficient.

The primary benefit of log analysis includes identifying unusual activities that may be early indications of failures, and aids in identifying and responding to incidents in a timely manner. Log analysis and reporting may also be used for auditing and verifying compliance when reports are used as artifacts.

Log Analysis Process

  • Data Collection: gathering messages from various sources
  • Data Indexing and parsing: Organizing logs to make them easier to search and categorize
  • Data Analysis: looking for patterns or anomalies, or looking for answers to specific questions
  • Monitoring: Keeping an eye on the logs to detect unusual or suspicious activity, or identify events and incidents, possibly generating alerts
  • Reporting: Summarizing the findings, may include charts and graphs and other visualizations, or may include comparative analysis of changes over time

Finding Value

There are typical challenges with log analysis:

  • Too many logs: Log entries can be extensive and can add up to many GB per day in a typical corporate infrastructure.
  • Inconsistent message format or content: Many applications and systems create log messages that are free-form and specific to the choices the creators of that application or system have made.
  • Missing or limited logs: In busy systems, logging may be disabled or limited due to resource strains, or to reduce system load or costs. That means some messages and events may not be logged at all, or logged events may be severely limited.

On the other hand, organizational information security goals and policies may be used to define what to look for in logs. As an example, if availability and uptime are of utmost importance, hardware status logs may be more valuable in an on-prem infrastructure, since they may correlate to the desired outcome, while the same logs may be completely irrelevant in a cloud deployment, where infrastructure is not under the direct control of an organization.

Additionally, a baseline needs to be established to identify what should be considered "normal" in order to recognize what would be "interesting" or "unusual".

Therefore, the first step in log analysis is to pare the incoming log data to only include the types of events that you care about, so you can more readily see when you get something out of the ordinary. To obtain value from the logs, you must ensure that:

  • Message details, quality, and integrity are assured
    • Configure your sources to send relevant information and at sufficient levels of details
    • Make sure that clocks are synchronized
    • Ensure consistency in implementation of infrastructure and configurations (for example, using DNS names, hostnames, or IP addresses)
    • Ensure that steps are taken to ensure that logs are protected against tampering (For example, intruders may tamper with logging services, or with the messages to hide their presence, or obscure the attack vector)
  • Messages are parsed correctly, so they are useful
    • Different applications, or different versions of the same application, may have varying structures or no structure in their messages

Examples of Interesting Events

  • Interesting network/router events:
    • access and authentication events, such as entering "enable" mode in a Cisco device or failed login events
    • Changes in interface statuses such as link states, addresses, or latency, or reachable routes
    • Firmware or configuration changes
  • Interesting Firewall events:
    • Access events, especially from remote networks
    • Administrative events such as log-in configuration changes, etc.
    • Filtered traffic data, what is being allowed and what is being blocked, and the frequency and prevalence of such events
  • System events
    • reboots, crashes (panics), shutdowns, access to single-user mode in some systems, etc
    • Hardware statuses such as memory and drive failures
    • Resource statuses, such as CPU utilization, memory consumption, storage consumption, etc.
    • Unplanned software or configuration changes, or unexpected open/listening ports
    • Access/authentication failure or success, and unusual sources of access
  • Database events
    • Access events and authentication, particularly administrative/root access, or application access from sources other than the application servers
    • Interactive client access vs. application back-end access
    • schema queries, invalid queries, and other indications of probing the database schema
    • changes to software and scripts that access the database
  • Web server events
    • Access permission issues (401 errors, excessive 404 errors, etc.)
    • Back-end errors (for example, 500 internal server errors, or database access errors)
    • unusual resource requests, for example, attempt to access php.info, .htaccess files (or any "dot" files), my.cnf, etc.
    • Web service process crashes
    • unexpected configuration or configuration changes
    • Malicious URL signatures, for example, injection attacks
    • Traffic volume spikes
  • Storage devices
    • Access logs must be from known hosts (especially in case of a SAN), and names must match the expected IP (in case of iSCSI) or WWN (World Wide Names for SAN). A mismatch may indicate a forgery attempt or other underlying infrastructure issue
    • Unusual commands sent to the storage device
    • Client and user access logs on NAS, based on the protocol (SMB for Windows sharing and NFS for Unix/Linux)

Log Analysis Tools

Local Log Analysis

If logs are stored in ASCII text format, generally any text processing utility can be used to look at the logs. In Linux/Unix, cat, grep, tail, more, and any utility that can sort and display text can be used in any combination as needed.
For Linux systems that use systemd, a common utility is journalctl (please refer to man journalctl in your Linux systems for all options and uses). This utility can also filter through and output logs based on keywords, priority, or facility/service names. This can be used in CLI manually or can be used as part of scripts to create desired reports.

Outside of the official documentation, you may find many articles and tutorials that you can use to practice and learn using this utility. Here are some examples of useful options:

  • journalctl -u apache should display any entries logged by the apache service unit
  • journalctl -p warning displays entries with priority level of warning
  • For troubleshooting boot and kernel/driver issues on a Linux system:
    • journalctl --list-boots to get a list of kernel boot logs available.
    • The IDs for each boot log start from zero (0) and go into negative numbers (-1, -2, etc), 0 indicating the current system boot up, and each negative number is a previous boot up before the current one
  • journalctl --since "YYYY-MM-DD HH:MM:SS" --until "YYYY-MM-DD HH:MM:SS" to view a time range of logs
  • To view live logs, you can use the -f option for "follow": journalctl <options_or matches> -f will display the logs live, useful for troubleshooting and live monitoring.

Central Log Analysis for Medium and Large Networks

Viewing logs in a central system is similar to viewing logs from a single system, with the only difference being the ability to specify a hostname to view logs from that host. Usually, the logs from many systems are going to become very large and storing them in ASCII text files may not be the most convenient to parse. Other data and file storage formats, such as databases (MySQL, PostgreSQL, Elastic, etc), XML, or JSON, are used.

Depending on the format of log storage, you can develop your own scripts and generate reports from the CLI, or use any programming or scripting language to automate log analysis. For example, if a MySQL database is used, the MySQL client or any other front-end client application may be used. The LogAnalyzer is an example of such an application written in PHP and can be deployed in Apache and NGINX web servers.

Here are some examples of MySQL queries to look at logs using mysql-client from bash:

sudo mysql <<<"select count(*) from Syslog.SystemEvents where FromHost = 'webhost' or FromHost = 'OPNsense';"

sudo mysql <<<"select count(*) from Syslog.SystemEvents where FromHost != 'loghost';" | more

sudo mysql <<<"select FromHost from Syslog.SystemEvents where FromHost != 'loghost';" | more

# list all MySQL user accounts:
sudo mysql <<<"select user from mysql.user;" | more

# Delete Records:
sudo mysql -u root <<< 'DELETE from Syslog.SystemEvents where FromHost = "laptop.home.arpa";'
sudo mysql <<<"select count(*) from Syslog.SystemEvents where FromHost = 'webhost' or FromHost = 'OPNsense';"

sudo mysql <<<"select count(*) from Syslog.SystemEvents where FromHost != 'loghost';" | more

# list all MySQL user accounts:

For enterprise environments, there is no shortage of commercial and open-source solutions. Traditional vendors may sell software licenses, services, or server appliances. More recent solutions can be self-hosted, subscription-based, or cloud-based. Additionally, there are ready-made utilities and services to convert from one format to another for use with external analysis tools. For example, logstash and filebeat are open-source utilities used to pull logs from one format (database or ASCII files) and convert them to other formats (JSON and XML) to be consumed by other applications. You can build your own solution by combining any available open-source tools to fit your exact needs.

Log Management and Maintenance

While the organizational security policy dictates some of the terms, in general, logs are not retained forever. In some cases, messages are only useful for a specific period of time, and their usefulness is diminished over time. Additionally, there are only so many resources available to dedicate to logs. It does not take a lot to assume that the cost of holding on to old logs may outweigh their value and benefits.

One method of maintaining logs is by rotating logs. Log rotation refers to discarding old logs in order to free up resources for new logs or to otherwise limit resources dedicated to log storage. Of course, the log retention and rotation policies vary from service to service and from application to application and must be based on organizational information security policy. In most systems, a logrotate utility or configuration is provided and must be configured accordingly.

There are automated (often commercial) tools that can automatically analyze logs, create reports and then discard the logs. This is a form of compression and is often done for ephemeral logs such as network flow logs. For example, we may be interested in overall bandwidth consumption, and keeping track of how much consumption came from each network or the internet, so we may use a tool such as "netflow" logs to extract this information from all network packet logs, then discard the detailed network packet and frame logs, and only retain what we are interested in.
