08 - AAA

Authentication, Authorization, and Accounting

Source: https://github.com/zonzorp/NETS1037/blob/main/Presentations/NETS1037-09-AAA.pdf

  • Authentication is the process of verifying the identity of an entity, whether that is a person,
    program, or machine
  • Authorization is the granting of permission to access resources, run programs, and save or
    change data
  • Accounting is the mechanism used to create, transmit, and store records describing activity

Authentication

Authenticating an entity does not grant privileges to the entity; it simply seeks to establish a reasonable level of confidence that the entity is who or what it claims to be.

Authenticating an entity can take several forms. An entity can provide one or more of the following to demonstrate the validity of its identity claim:
• A login name or identifying string of characters, usually accompanied by a password or secret string associated with
that name
• A certificate signed by a trusted authority
• A valid response to a challenge issued when an identity is claimed
• Physical or virtual possession of auxiliary information such as smart cards, dongles, or one-time code generators
• Biometric data unique to the entity when the entity is biological
• One or more pieces of information theoretically distinct to that device, such as the Ethernet address

Authorization

Authorization is the granting of permission to perform tasks or access resources based on one
or more attributes of the entity requesting authorization. The primary attribute is the requester's identity; therefore, authentication precedes authorization in the normal processing of requests when the requester's identity is relevant. When the authorized activity requires identification but no requester identity is implemented, session identifiers are often used. Additional attributes may also be used, such as method of access, time of access, protocols used, other parties involved, and attributes of the requested resources.

Some examples of implementations include:

  • Resource and protocol without authentication
    • public web servers provide authorization to clients to access resources based on the protocol and the resources requested
  • Identity-based authorization after authentication
    • RBAC example: students and instructors authenticate to the same system, but authorization allows access to different resources on the system based on the specified roles
  • Session-based authorization after authentication
    • session cookies are stored in the web browser and are presented to the web server instead of re-authentication
  • Time-based authorization
    • Access to an exam is granted only within a certain period of time
  • Multi-Attribute Authorization
    • Banking App: requires authentication, requires a trusted device requires MFA, requires HTTPS with TLS 1.3, requires an allowed region (based on the geolocation of the client IP address)

Accounting

Accounting is the process of tracking, recording, and auditing user activities and system resource usage to ensure accountability and support security monitoring. Accounting requires capturing information which describes the activity. At a minimum, access accounting typically includes authentication info, authorization(s) granted, access method and location, timestamps, and when that activity changed (e.g. login/logout, change privileges, etc.).
Accounting may include statistical or detailed data about what was done while authorization was granted. The detail level is usually related to some type of activity we are interested in summarizing, such as total network traffic moved, or disk i/o caused, etc.
Auditing is a highly detailed account of the level of specifically what was accessed, when, and the results of the access.

AAA Implementation and Solutions

AAA may be implemented using a single protocol or a single program or software package, or may be split between various systems. Windows Active Directory is an example of a non-integrated solution that handles authentication and authorization, but does not handle accounting on its own. The events are logged; however, log analyzers must be used to correlate events to make them useful. Additionally, Windows Active Directory does not record any detailed access and resource usage and must be combined with other tools such as Microsoft Sentinel or 3rd party products.

Standalone servers provide login authentication locally. Authorization is done by user group membership and login shell, and accounting via tools such as syslog or direct service and application logging. Clients in small organizations typically use local per-host authentication/authorization, may replicate authentication/authorization between machines, or may use distributed authentication/authorization and usually ignore accounting.
SMEs (Small and Medium Businesses) often implement Microsoft Active Directory only, using centralized authentication and ignoring accounting. Larger organizations use distributed AAA solutions via servers such as Cisco Access Control Server (ACS) accessing backend AAA services for edge access, and LDAP-based solutions for user access. Distributed AAA services are often provided by RADIUS or TACACS+ servers (Cisco solution).

Entity Definitions and Storage

A basic element of authentication is to determine how to identify entities in a useful way. Users may be defined by their user accounts in various ways (for example, by UID in Linux/Unix systems). Machines or devices can be identified by name, address, network, etc.
These definitions are stored in a database. The database can be simple flat files (such as /etc/passwd and /etc/shadow), in distributed secure access databases (such as Active Directory or OpenLDAP), or in a database platform (such as MySQL/MariaDB). The identity stores and their access mechanisms are collectively referred to as Directory Services. Identity lookup programs (such as RADIUS) can be used with plug-ins to access these data sources.

OpenLDAP

OpenLDAP (Open Lightweight Directory Access Protocol) is an open-source implementation of the LDAP protocol used for managing and accessing directory information (such as users, groups, and authentication data) over a network. It includes a Directory Information Tree (DIT) and client tools to access that DIT. Entries to DIT are made in LDAP Data Interchange Format (LDIF).

In Linux systems, the slapd package is the server and ldap-utis are the associated utilities. Changes to the database are natively done via CLI tools. There are GUI/WebUI tools available; for example, Webmin is popular, but there are many more, such as Apache Directory Studio as part of the The Apache Directory™ Project, phpLDAPadmin, LDAP Account Manager (LAM), FusionDirectory, etc.

Network and Devices

AAA can be applied to systems (devices and software), as well as users. For network edge user authentication, RADIS and TACACS are two protocols aimed at providing AAA to network devices or remote connections. TACACS is typically implemented to support Cisco devices in the network, and RADIUS is for most other AAA situations. Either one can use flat files, LDAP, or other directory services to provide user/machine definitions or store them in service-specific databases.

TACACS (Terminal Access Controller Access-Control System) is a network authentication protocol used to control access to network devices such as routers and switches. TACACS+ is the modern and widely used version, which adds full encryption and more granular authorization. This protocol is mostly used for administrator access to networking equipment and is almost never used for user access.

Source: The FreeRADIUS Technical Guide:
RADIUS, which stands for “Remote Authentication Dial-In User Service”, is a network protocol for remote user authentication and accounting. RADIUS protocol serves three primary functions:

  • Authenticates users or devices before allowing them access to a network
  • Authorizes those users or devices for specific network services
  • Accounts for and tracks the usage of those services

source: https://en.wikipedia.org/wiki/TACACS#Comparison_with_RADIUS
Here is a quick comparison between RADIUS and TACACS+ protocols:

Feature TACACS+ (RFC 8907) RADIUS
Protocol Type Cisco-developed (proprietary, but widely supported) Open standards (IETF)
Transport Protocol TCP (port 49) UDP (ports 1812, 1813, 1814 and 18120 by default), can use TCP (RFC6613) and TLS (RFC 6614), and DTLS (RFC7360)
Encryption Can encrypts entire payload, or use plain-text + IP Filtering for security Encrypts password field
AAA Separation Fully separates authentication, authorization, accounting (more granular, down to commands) Combines authentication + authorization
Use Case Network device administration (routers, switches) User/Device Network access (Wi-Fi, VPN)
Vendor Support Primarily for Cisco switches and routers, only for administrator access to networking equipment Widely supported (multi-vendor), can be modified to extend functionality or support any type of security technology
Deployment Robust functionality for administrator authentication and command authorization, but is not used for authenticating end-user access to networks Offers minimal functionality for administrator authentication and command authorization, while offering strong support (and is widely used) for end-user authentication, authorization, and accounting

RADIUS Protocol

Source: The FreeRADIUS Technical Guide:
RADIUS Components:

Component Name Functions Examples
User/Device Requests access to the network. Laptop, ADSL Modem, VOIP Phone
Network Access Server (NAS) Provides access to the network for the user/device (gateway) Switch, Wireless Access Point, DSLAM (Digital Subscriber Line Access Multiplexer), VPN Terminator
Authentication Server Receives authentication requests from the NAS, Returns authentication results to the NAS, Optionally requests user and configuration information from the database or directory, May return configuration parameters to the NAS, Receives accounting information from the NAS FreeRADIUS, Radiator, NPS (Microsoft's RADIUS Server), ACS (Cisco Secure Access Control Server)
Data Store Optional database or directory with user authentication and authorization information. RADIUS server communicates with the data store using DB API or LDAP. SQL Database, Kerberos Service Server, LDAP Directory

source: https://en.wikipedia.org/wiki/RADIUS#Authentication_and_authorization

  • The user or machine sends a request to a Network Access Server (NAS) to gain access to a particular network resource using access credentials.
  • NAS sends a RADIUS Access Request message to the RADIUS server, requesting authorization to grant access via the RADIUS protocol.
  • RADIUS server checks that the information is correct using authentication schemes such as PAP, CHAP or EAP.
    • The user's proof of identification is verified, along with, optionally, other information related to the request, such as the user's network address or phone number, account status, and specific network service access privileges
    • The RADIUS server has no control over the content of the request that the NAS sends
  • The RADIUS server returns one of three responses to the NAS:
    • Access Reject: The user is unconditionally denied access to all requested network resources
    • Access Challenge: Requests additional information from the user, such as a secondary password, PIN, token, or card.
      • Access Challenge is also used in more complex authentication dialogues where a secure tunnel is established between the user's machine and the RADIUS Server in a way that the access credentials are hidden from the NAS
    • Access Accept: The user is granted access.
  • Authorization attributes are conveyed to the NAS stipulating the terms of access to be granted
  • The NAS will send an "Accounting Start" packet to the RADIUS server for Accounting
    • The "Start" records typically contain the user's identification, network address, point of attachment and a unique session identifier.
  • The NAS will periodically send an "Interim Update" record to the RADIUS server
    • "Interim" records convey the current session duration and information on current data usage
  • The NAS issues a final Accounting "Stop" record to the RADIUS server when the connection is terminated, or the session ends
    • information on the final usage in terms of time, packets transferred, data transferred, reason for disconnect and other information related to the user's network access

For more detailed information, see the Wikipedia article on RADIUS.

BlastRADIUS Attack and Protocol Vulnerability (CVE-2024-3596)

Source: https://www.blastradius.fail/
The RADIUS protocol predates modern cryptographic guarantees and is typically unencrypted and unauthenticated. The RADIUS protocol uses UDP by default and passwords are encrypted using MD5 hash function. The Blast-RADIUS attack allows a man-in-the-middle attacker between the RADIUS client and server to forge a valid protocol accept message in response to a failed authentication request. This forgery could give the attacker access to network devices and services without the attacker guessing or brute forcing passwords or shared secrets.

  • The attacker does not learn user credentials.
  • The attacker needs access to the RADIUS network
  • The attacker must compute MD5 hash and find collisions successfully within the configured timeout (usually 30 to 60 sec by default)

The detection of the attack requires correlating Access-Rejects messages logs on the RADIUS server and Access-Accepts on the RADIUS client.

The mitigation involves a change in the RADIUS implementations to require sending Message-Authenticator attributes for all requests and responses. This requirement is being added to the IETF RFC to be made standard. FreeRADIUS and other vendors may have already implemented such requirements (see https://www.inkbridgenetworks.com/blastradius).

The long-term solution is to use an authenticated and encrypted channel for RADIUS such as TLS or DTLS instead of the plain UDP. Additionally, it is expected that most enterprises will route RADIUS traffic to the server over a separate management network to reduce the likelihood of access to RADIUS traffic.

FreeRADIUS

FreeRADIUS is the most popular and the most widely deployed open-source RADIUS server in the world. FreeRADIUS is designed to be modular, extensible, and scalable. It provides all aspects of AAA and supports additional plug-ins to link to other databases and services such as MySQL, PostgreSQL, LDAP, or Microsoft Active Directory.

Linux Installation and Configuration:

  • Installation of the freeradius package also automatically generates the required server certificates.
  • Listens on UDP ports 1812, 1813, 1814 and 18120 by default
  • Configuration is in /etc/freeradius
    • radiusd.conf is the primary configuration file for the service
    • clients.conf defines which authenticators can access the server and what the shared secrets are for them
    • The users file defines who can authenticate and what their authorizations are
    • acct_users defines the accounting records to keep

An example of extensibility is using RADIUS authentication to log into a Linux system locally or via SSH. This is not common; however, it demonstrates using an existing RADIUS server to authenticate and authorize a user.

  • install libpam-radius-auth package (installs pam_radius module under /lib/security)
  • Configure your users in the /etc/freeradius/users file
  • Add pam_radius.so to the authentication methods in the /etc/pam.d/application configuration files
  • Give consideration to how your RADIUS traffic is secured if your RADIUS server is not localhost (can be partially encrypted, use TLS with certificates, or tunnelled)
