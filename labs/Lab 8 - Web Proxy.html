Forward Proxy Using Squid

A forward web proxy is a server that sits between client devices (users) and the internet. Instead of users connecting directly to websites, their requests go through the proxy first. The proxy can enforce security policies, filter web content, log user activity, anonymize client IP addresses, cache frequently accessed content to improve performance, and control internet access for users within an organization.

On your webhost VM:

  • Install squid-openssl or squid package using apt.
    • The squid-openssl package is compiled with SSL/TLS optional tools required for transparent MITM proxy, while the squid package does not support HTTPS MITM interception. The configuration files are the same for both packages.
sudo apt update && sudo apt upgrade -y
sudo apt install squid-openssl

Reverse Proxy

A reverse web proxy is a server that sits in front of one or more web servers and handles incoming requests from users on the internet. Instead of clients connecting directly to the application server, they connect to the reverse proxy, which can provide load balancing, SSL/TLS termination, security protection, caching, and application routing. This improves performance, availability, and security while hiding the internal architecture of the backend servers from internet users, or can act as a gateway for providing access to resources between networks.

Here is an example of web servers behind a NAT/Firewall:

The following is another example of a network architecture where the two LAN networks are isolated. LAN 2 access, therefore, all client access to LAN 1 is blocked. The firewall allows the traffic from the proxy to flow to LAN 1 (allow list), while blocking all other traffic. Security can be further layered by only allowing traffic from the proxy on the web servers. Clients access the resource through the proxy. From the client's perspective, the resources are seemingly hosted on the proxy.

Reverse proxies such as NGINX, HAProxy, Azure Application Gateway, Azure Front Door, and Apache HTTP Server are commonly used to provide scalability, security, and reliability for internet-facing applications. In this lab, you should already have Apache installed on your webhost. We will enable the required modules to experiment with reverse proxy on Apache.

Add the following configuration to the virtualhost in your /etc/apache2/sites-available/000-default.conf site file:

  • disable forward proxying
  • pass the resource /escape/ to http://httpforever.com/
  • remove the Accept-Encoding header from the request because the proxied site does wonky things with the response encoding
  • correct the URL for HTTP responses
  • correct embedded links which may be in the returned HTML documents
ProxyRequests Off
ProxyPass /escape/ http://httpforever.com/
<Location /escape/>
      ProxyPassReverse /
      ProxyHTMLEnable On
      ProxyHTMLURLMap http://httpforever.com/ /escape/
      ProxyHTMLURLMap / /escape/
      RequestHeader unset Accept-Encoding
</Location>
ProxyRequests Off
ProxyPass /escape/ http://httpforever.com/
<Location /escape/>
      ProxyPassReverse /
      ProxyHTMLEnable On
      ProxyHTMLURLMap http://httpforever.com/ /escape/
      ProxyHTMLURLMap / /escape/
      RequestHeader unset Accept-Encoding
</Location>

Reload apache2 to get it to read in the changes

sudo apachectl graceful
sudo apachectl graceful

Set up simple content and virus management

  1. On your proxyhost already running squid, download and install e2guardian and clamav-daemon
sudo apt update
sudo apt-get install e2guardian clamav-daemon
sudo apt update
sudo apt-get install e2guardian clamav-daemon
  1. Edit /etc/e2guardian/e2guardian.conf:
    • Uncomment the contentscanner that enables clamdscan.conf
    • Set the daemongroup group name to clamav
    • check that the maxcontentramcachescansize is set to at least the same as the maxcontentfiltersize but not bigger than maxcontentfilecachescansize
  2. Edit /etc/e2guardian/contentscanners/clamdscan.conf:
    • Ensure clamdudsfile is set to the same as the filename specified in /etc/clamav/clamd.conf for the LocalSocket
      • Verify manually or use something like grep 'LocalSocket ' /etc/clamav/clamd.conf && grep clamdudsfile /etc/e2guardian/contentscanners/clamdscan.conf
      • They should be the same by default because /run and /var/run are linked. You can verify using ls -l /var
  3. Restart the e2guardian service, and start the clamav-daemon, which may not be running
  4. Open the e2guardian proxy port 8080 in your firewall
    sudo systemctl restart e2guardian
sudo systemctl start clamav-daemon
sudo ufw allow 8080/tcp
    sudo systemctl restart e2guardian
sudo systemctl start clamav-daemon
sudo ufw allow 8080/tcp
  5. Set the http_proxy variable to your proxyhost port 8080 so that e2guardian and clamav can intercept and inspect your web requests: export http_proxy=http://proxyhost:8080
  6. Run apt update to verify you can retrieve valid websites using the e2guardian and clamav proxy
  7. Try using wget to retrieve the 4 different eicar virus test files to verify you are blocking them properly. Make sure to enter these tests individually and observe the output:
