Lab 04 - SNMP

The purpose of this lab is to create an SNMP monitoring station and use it to monitor the loghost, webhost, and OPNsense router already installed on our virtual networks.

DNS Overrides on OPNsense

Before you begin, start your opnsense router and log into the web UI and add the following DNS override:

  • Navigate to: Services > Unbound DNS > Overrides
  • Add the following overrides:
    • Host: nmshost
    • Domain: home.arpa
    • IP Address: Your Host-only network with host address 5, for example: 192.168.60.5
    • Any Description for your reference, for example, The SNMP host

VM Creation and OS installation

Please keep in mind that you will have your opnsense router and loghost running at the same time, so make sure to monitor your host resources. Exit background tasks and unnecessary programs on your host computer to free up resources if needed.

Create a new VM for Ubuntu Server and attach the NIC to your Host-only Network.

  • Start with at least 2 CPUs
  • Start with at least 1GB
  • Set the disk to be at least 20GB. You can resize this later if needed; however, additional steps will be needed to resize the partition and volume in Linux.

Install Ubuntu Server LTS (download the latest .iso image from ubuntu.com) with the following configuration during setup:

  • Configure your NIC with Manual Configuration:

    • Use the Static IP address from your network with the host address set to 5 (for example, 192.168.60.5)
    • The Gateway and Name Server should be set to your OPNsense router IP Address you had created in the previous lab (for example, 192.168.60.1)
    • Use the search domain home.arpa

  • Use the entire disk (can use the default settings)

  • Under Profile Configuration, enter the following:

    • Set Your server name (system hostname) to nmshost
    • Add a user account for yourself (your first name) and configure a password

  • Under SSH Configuration choose to Install OpenSSH server

  • You do NOT need to install any additional software or packages.

  • Once the installation is complete, reboot the server and disconnect the installation .iso image from the optical drive and hit return when prompted.

If your environment is configured correctly and installation is successful, you should be able to log in through console or SSH, and your server should have access to the internet for downloading and installing updates through your OPNsense router. You can attempt to update and upgrade your system using sudo apt update && sudo apt upgrade -y. Additionally, attempt to ping nmshost from loghost using ping nmshost to verify the DNS configuration and IP configurations are correct.

Configure loghost to Allow SNMP

  • Install the snmpd package on loghost, configure the agentaddress to accept remote connection, and open firewall port as follows:
sudo apt install snmpd
sudo ufw allow 161/udp
sudo apt install snmpd
sudo ufw allow 161/udp
  • Modify the snmpd.conf (for example, using sudo nano /etc/snmp/snmpd.conf) as follows:
# agentaddress: The IP address and port number that the agent will listen on.
#   By default the agent listens to any and all traffic from any
#   interface on the default SNMP port (161).  This allows you to
#   specify which address, interface, transport type and port(s) that you
#   want the agent to listen on.  Multiple definitions of this token
#   are concatenated together (using ':'s).
#   arguments: [transport:]port[@interface/address],...

#agentaddress  127.0.0.1,[::1]     # Commented out the default configuration
agentaddress  udp:161,[::1]:161    # We are currently not using IPv6

#   By default the agent listens to any and all traffic from any
#   interface on the default SNMP port (161).  This allows you to

#   want the agent to listen on.  Multiple definitions of this token

#   arguments: [transport:]port[@interface/address],...

#agentaddress  127.0.0.1,[::1]     # Commented out the default configuration

  • Restart the SNMP Daemon:
sudo systemctl restart snmpd

# Verify the service is started
sudo systemctl status snmpd

# optionally verify that the service is listening on port 161 as expected
sudo ss -utlpn

  • Verify you can access the SNMP tree from nmshost using version 1 community public. Test this by using snmpstatus and snmpwalk utilities to access loghost from nmshost
# On the `nmshpst`, you may need to install snmp first: 
sudo apt install snmp
snmpstatus -v 1 -c public loghost
snmpwalk -v 1 -c public loghost | wc -l # counting the number of items exposed
# On the `nmshpst`, you may need to install snmp first: 
sudo apt install snmp
snmpstatus -v 1 -c public loghost
snmpwalk -v 1 -c public loghost | wc -l # counting the number of items exposed
  • On the loghost, enable full SNMP tree read-only access for community public to be defined as your network instead of the default system-only option. Change the rocommunity setting in /etc/snmp/snmpd.conf and use your Host-only Network address:
# Read-only access to everyone to the systemonly view
# rocommunity  public default -V systemonly
rocommunity public 192.168.60.0/24   # Use your host-only network IP address
# Read-only access to everyone to the systemonly view
# rocommunity  public default -V systemonly
rocommunity public 192.168.60.0/24   # Use your host-only network IP address
  • Restart the service for changes to take effect: sudo systemctl restart snmpd
  • Run snmpstatus from nmshost again to verify you can now retrieve more OIDs than you could before
# On the `nmshost`, check to see if the numbers and states changed compared to earlier
snmpstatus -v 1 -c public loghost
snmpwalk -v 1 -c public loghost | wc -l  # Counting the number of items exposed
# On the `nmshost`, check to see if the numbers and states changed compared to earlier
snmpstatus -v 1 -c public loghost
snmpwalk -v 1 -c public loghost | wc -l  # Counting the number of items exposed

Set up webhost for SNMP Monitoring

  • Install the snmpd daemon on webhost, allow SNMP access through your ufw firewall, and modify /etc/snmp/snmpd.conf to allow access and enable full SNMP tree access as you did on loghost:
# Connect to webhost or start a new SSH session from your host system
ssh webhost

# On webhost
sudo apt install snmpd
sudo ufw allow 161/udp
sudo nano /etc/snmp/snmpd.conf
sudo systemctl restart snmpd
# Connect to webhost or start a new SSH session from your host system
ssh webhost

# On webhost
sudo apt install snmpd
sudo ufw allow 161/udp
sudo nano /etc/snmp/snmpd.conf
sudo systemctl restart snmpd
  • Verify you can access the SNMP tree from nmshost using v1 community public:
# On the `nmshost`
snmpstatus -v 1 -c public webhost
snmpwalk -v 1 -c public webhost | wc -l
# On the `nmshost`
snmpstatus -v 1 -c public webhost
snmpwalk -v 1 -c public webhost | wc -l

Add MIBs to nmshost To Have More Descriptive OIDs

The MIBs descriptors may be copyrighted, and they are not enabled on Linux systems by default. You need to add additional repositories and install additional packages to translate OIDs from numbers to a more human-readable format. The following is an example of before and after enabling the feature:

Using OIDs only After enabling MIBs
iso.3.6.1.2.1.1.3.0 = Timeticks: (19935) 0:03:19.35 DISMAN-EVENT-MIB::sysUpTimeInstance = Timeticks: (19935) 0:03:19.35

From The Debian Package: snmp-mibs-downloader (1.7) [non-free] description:

The snmp-mibs-downloader "package ships the IETF RFCs containing MIB files and a script which extracts them to be used by Simple Network Management libraries. The script can be used to update some MIBs to the latest version or to download extra vendor MIBs. ... These MIBs can be useful for ... translating the received information into human readable text."

  • Install the snmp-mibs-downloader package on nmshost:
sudo apt install snmp-mibs-downloader
sudo apt install snmp-mibs-downloader
  • Configure nmshost to use the mibs when running SNMP commands. Modify the /etc/snmp/snmp.conf file and comment out the mibs : line to see descriptions instead of OIDs (Object Identifiers):
sudo sed -i -e 's/(^mibs)/#\\1/' /etc/snmp/snmp.conf
# or simply use nano and comment out (using #) the line so that:
#mibs :
sudo sed -i -e 's/(^mibs)/#\\1/' /etc/snmp/snmp.conf
# or simply use nano and comment out (using #) the line so that:
#mibs :
  • Retest using snmpwalk to verify you now see and can use descriptive OIDs on nmshost:
# The following will fail since OPNsense would not have the SNMP module installed yet:
snmpwalk -v 1 -c public nmshost | more
# The following should succeed: 
snmpwalk -v 1 -c public loghost | more
snmpwalk -v 1 -c public webhost | more
# The following will fail since OPNsense would not have the SNMP module installed yet:
snmpwalk -v 1 -c public nmshost | more
# The following should succeed: 
snmpwalk -v 1 -c public loghost | more
snmpwalk -v 1 -c public webhost | more

Add an SNMPv3 User to Secure SNMP Access and Transport

You can use SNMPv3 Options or Configuring SNMPv3 section of O'Reilly book as a reference guide to adding users for SNMPv3 connections.

  • Add an SNMPv3 user to /etc/snmp/snmpd.conf on the loghost and webhost machines, which has authentication (using SHA) and encryption (using AES) enabled
# Must have root access
# Notice the credentials in plain text in the config file:
echo "createUser authPrivUser SHA password123 AES password456" >>/etc/snmp/snmpd.conf
# Restart service:
systemctl restart snmpd
# Are the credentials stored in Plain Text here? 
grep usmUser /var/lib/snmp/snmpd.conf
# Must have root access
# Notice the credentials in plain text in the config file:
echo "createUser authPrivUser SHA password123 AES password456" >>/etc/snmp/snmpd.conf
# Restart service:
systemctl restart snmpd
# Are the credentials stored in Plain Text here? 
grep usmUser /var/lib/snmp/snmpd.conf
  • Test that you can use authentication and encryption with snmp command line tools from nmshost to loghost (this example has username authPrivUser with password password123 for SHA login and secret password456 for AES encryption)

Screenshot this command and its results, including your command prompt

snmpstatus -u authPrivUser -a SHA -A password123 -x AES -X password456 -v 3 -l authPriv loghost
snmpstatus -u authPrivUser -a SHA -A password123 -x AES -X password456 -v 3 -l authPriv loghost
  • Test that you can use authentication and encryption with snmp command line tools from nmshost to webhost (this example has username authPrivUser with password password123 for SHA login and secret password456 for AES encryption)

Screenshot this command and its results, including your command prompt

snmpstatus -u authPrivUser -a SHA -A password123 -x AES -X password456 -v 3 -l authPriv webhost

Basic Monitoring with LibreNMS

LibreNMS is a pre-configured fork of Observium for common system and network monitoring tasks. It is a webapp, which can be installed on apache2 or nginx running on Linux. We will use it to demonstrate gathering data with SNMP for a graphical presentation on a network monitoring station. We will use librenms because it is pre-configured.

Install librenms on nmshost:

Appendix

How to Enable SNMP on OPNsense

  • Navigate to System, Firmware and Plugins
  • Make sure that the Show community plugins checkbox is enabled at the top right of the table header
  • search for os-net-snmp and install by clicking the + icon

  • Once the installation is completed, refresh the page and navigate to Services, and Net-SNMP.
  • Enable the SNMP service under the General tab.
  • Use the SNMPv3 tab to add and configure SNMPv3 users
