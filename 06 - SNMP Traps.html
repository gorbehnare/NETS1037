06 - SNMP Traps

Proactive Monitoring

Having staff observe all logs and attempting to identify and sort through actionable incidents is impractical and very costly. Most enterprise systems and devices have a reporting mechanism to report and bring attention to issues. These alerts and notifications can be visual or audible for local proximity notification use cases, but can also be communicated to other systems and staff over communication channels such as telephone and mobile networks (voice calls, pagers, and SMS), and IT networks (integration with instant messaging systems, push notifications, emails, etc.).

The SNMP Trap

When SNMP is used for logging and monitoring, normally the NMS (Network Management Station) gathers configuration and status data by polling the agents (for example, the NMS reaches out to agents with SNMP GET requests or walking OID on each device). However, agents can be configured to proactively send unsolicited messages to NMS in the form of SNMP Traps. This is done for two purposes:

  1. To communicate status changes, warnings, errors and failures
  2. To reduce network bandwidth in very large monitor networks (polling hundreds of devices with thousands of OIDs each can be resource-intensive)

SNMP Traps enable an agent to notify the management station of significant events by way of an unsolicited SNMP message. Trap PDUs include the current sysUpTime value, an OID identifying the type of trap and optional variable bindings. Destination addressing for traps is determined in an application-specific manner, typically through trap configuration variables in the MIB. The format of the trap message was changed in SNMPv2, and the PDU was renamed SNMPv2-Trap. (source: https://en.wikipedia.org/wiki/Simple_Network_Management_Protocol)

In SNMP v1, there are two types of traps:

  • Generic Traps (defined in RFC 1215, and are only 6 standard traps: coldStart, warmStart, linkDown, linkUp, authenticationFailure, and egpNeighborLoss)
  • Enterprise-specific Traps (vendor custom-defined traps)

For a management system to understand an enterprise-specific trap sent to it by an agent, the management system must know what the object identifier (OID) defines. Therefore, it must have the MIB for that trap loaded. That means for every monitored device, the vendor-specific MIB must be obtained and added to the NMS.

The SNMP v2c removes the SNMP v1 Trap types and sets the type as NOTIFICATION. The SNMP v3 further adds authentication and encryption to the SNMP v2c traps. Since SNMP Traps for v1, v2c, and v3 are incompatible in both the message format and the protocol operations, the NMS must overcome incompatibility issues in a mixed protocol environment. The RFC 3584 (Coexistence between Version 1, Version 2, and Version 3 of the Internet-standard Network Management Framework) defines coexistence strategies of using proxy agents to translate and forward, or using multi-lingual network-management systems.

Installation and configuration

In a Linux system, the snmptrapd is a separate package to be installed and configured. The main configuration is at /etc/snmp/snmptrapd.conf. Additionally, need to enable snmptrapd in the /etc/default/snmpd. See the SNMP trap lab for more details.

SNMP Traps Operation, Limitation, and Consideration

  • An SNMP agent can send a trap message on UDP port 162 to a trap receiver, which then takes some action based on the information contained in the trap (and the OID and MIB).
    • All vendor-specific and important MIBs must exist and be configured correctly for correct MNS operations
  • Since SNMP Traps are sent over UDP, they are subject to loss and errors. UDP is transmitted without any acknowledgement and as a best-effort delivery type.
    • The transport layer (UDP) does contain error-checking information, but malformed or corrupted packets are simply discarded
    • Some SNMP implementations require acknowledgements from managers; however, this is not part of the standards.
  • SNMP can be used to attack networks:
    • SNMP can be used for reconnaissance (walk the OIDs, and the device will spill all it is configured to display)
    • SNMP can be used to configure devices or modify configurations remotely
    • Credentials in v1 and v2c can be trivially sniffed
      • SNMP v1 credentials are plain text
      • SNMP v2 adds optional MD5 hashing (must be configured)
    • SNMP v3 credentials can be subjected to dictionary attacks and brute-force when weak passwords and algorithms are used
    • SNMP can be subjected to DoS
  • Best practices. controls and mitigations (for example, see CISA recommendations at https://www.cisa.gov/news-events/alerts/2017/06/05/reducing-risk-snmp-abuse):
    • Use SNMP v3 when available and use longer and more complex passwords and higher ciphers (use authPriv)
    • Limit SNMP traffic to/from specific IP addresses or networks using firewalls and ACLs
    • Physically separate the management network from production networks
    • If SNMP needs to be transmitted across networks, combine with other encryption and tunnelling such as TLS, DTLS, SSH, VPNs, etc.
  • SNMP Security: The Net-SNMP.org has a tutorial and summary table available at Net-SNMP Tutorials/SNMP Security:
Protocol Description
SNMPv1 or SNMPv2c Provides NO SECURITY at all. Supported by everything.
SNMPv3 with USM The original SNMPv3 security model. Supported by most devices.
SNMPv3 with TLS or DTLS Tunneling SNMP over TLS and DTLS. Hopefully, it will be supported by most devices soon.
SNMPv3 over SSH Tunnelling SNMP over SSH (not 100% supported in Net-SNMP)
SNMP over Kerberos Securing SNMP with Kerberos (even less supported)
